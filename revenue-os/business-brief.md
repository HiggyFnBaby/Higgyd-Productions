# Revenue OS — Business Brief

**What this is, in one sentence:** a multi-agent Revenue Operating System,
built as a real Claude Code project (not a chat answer), that turns raw
market signal into a working sales conversation — for Derrick's own app
portfolio first, and as a sellable SaaS/CRM for other entrepreneur-creators
second.

If you're new to some of these words: a **subagent** is a specialist Claude
Code persona with its own job and its own instructions — think of it as
hiring four different specialists instead of asking one generalist to do
everything. A **runbook** is the step-by-step manual for how a human actually
operates the system day to day. Neither of those ideas requires you to write
code — you're directing specialists, the way you already direct workflows in
a no-code tool.

## The core idea: money is not a tool, money is in systems

It's tempting to think the thing that makes money is a tool — the CRM, the
landing page builder, the AI chatbot, the automation platform. It isn't.
Tools produce *activity*. A CRM with no clear offer just stores contacts. An
AI agent with no distribution just answers questions. Activity without a
system connecting it to a paying customer is just motion, not revenue.

**A system is a repeatable path from "someone has a real problem" to "someone
paid you to solve it."** That path has four links, and each one is a
specialist job:

1. Find proof a problem is real and people already want it solved
   (**market-signal-researcher**).
2. Turn that proof into something specific and sellable — what, to whom, at
   what price (**offer-architect**).
3. Turn that offer into words that make the right buyer stop scrolling
   (**content-angle-strategist**).
4. Turn attention into a paying customer through a designed conversation, not
   a hope-they-buy landing page (**conversation-system-builder**).

The tool (your CRM, your automation platform, your app) is where step 4's
output gets *implemented*. The tool never generates the first three steps —
only the system does. This is why "add a payment button" rarely works on its
own, and why this project builds the system first, tool integration second.

## How the four agents connect (the workflow-to-revenue chain)

```
market-signal-researcher
        │  (signal report: what's the proven pain?)
        ▼
  offer-architect
        │  (offer brief: what are we selling, to whom, at what price?)
        ▼
content-angle-strategist
        │  (angle set: what makes the right buyer stop and read?)
        ▼
conversation-system-builder
        │  (conversation blueprint: the actual DM/CRM flow that closes)
        ▼
  IMPLEMENTED IN THE CRM/AUTOMATION TOOL  ──▶  REVENUE
        │
        └──── results (what closed, what got objected to) feed back
              into market-signal-researcher as new signal
```

Each arrow is a handoff, not a guess — the output of one agent is the literal
input to the next. The tool only enters the picture at the very last step,
after the system has already decided what the conversation should say and
why. Revenue isn't generated by the CRM automation firing — it's generated by
the automation being built on top of a proven signal → offer → angle chain.
The full operating sequence is in `runbooks/revenue-agent-runbook.md`.

## The v1 application (`app/`)

The idea above is now a real, working multi-tenant Next.js + Prisma app, not
just planning docs — see `app/README.md` for how to run it. It builds and
typechecks clean (verified 2026-07-09). Summary of what's real:

- **Auth & multi-tenancy:** email/password signup creates a Workspace +
  owner Membership; all data is scoped to `workspaceId`.
- **Pipeline board:** leads move through SIGNAL → OFFER → ANGLE →
  CONVERSATION → WON/LOST — the CRM pipeline literally *is* the Revenue OS
  chain, not a generic CRM with agents bolted on.
- **Classic automation:** every stage change auto-creates a next-action Task
  (`src/lib/automations.ts`) — a lead never just goes quiet.
- **AI-agent automation:** the "Run agent" button on a lead calls the Claude
  API using the *actual* `.claude/agents/*.md` files as system prompts
  (`src/lib/agents.ts` reads them directly — the app and the planning docs
  can't drift apart, they're the same files).
- **Billing:** provider-agnostic interface (`src/lib/billing/types.ts`) with
  a working Stripe adapter (checkout + webhook). Paddle/LemonSqueezy can be
  added later as a second adapter without touching call sites.

v1 shortcuts worth knowing about: no OAuth login yet (email+password only),
no team invites (one user per workspace), stage change is a dropdown not
drag-and-drop. None of these block using it — see `app/README.md` for the
full list and for the 3 things needed to actually run it (a Postgres DB, an
Anthropic API key, a Stripe account).

### Paywall enforcement + marketing page (added 2026-07-23)

The gap that made this "not sellable yet" — paying didn't actually unlock
anything — is closed:

- Every workspace gets a 14-day free trial (`trialEndsAt`, set at signup, no
  credit card needed).
- `src/lib/access.ts` is the single place "does this workspace get to use
  the app" is decided (active subscription OR still within trial).
- That check runs in **two** places, not just one: the dashboard pages
  (redirect to `/dashboard/billing` if expired) AND the API routes that cost
  money or create usage — creating leads, changing stage, and especially
  running an agent (a real Anthropic API charge per call). Gating only the
  UI would have left the API callable directly after a trial expired; both
  are covered.
- `/dashboard/billing` is the one page that never gates — it's where an
  expired workspace lands to see why and upgrade.
- The homepage is now an actual pitch: the four-agent chain explained,
  pricing card, trial CTA — not just two login links.
